## 設計意図

- app/ 内の見通しを良くするため、 `/` を `/search` にリダイレクトする
- 検索キーワードやページ番号を URL クエリパラメーターで管理することで、ブラウザの戻る・進むで状態を復元できるようにする
- 詳細画面は Server Component で実装することで、パフォーマンスの向上と実装のシンプルさを両立する

## 採用技術意図

- GiHub API は REST を利用
  - 今回は2画面しかなく、複数のエンティティにまたがったデータの取得が不要であるため、シンプルな実装で済む REST を採用
- UI ライブラリに CSS Modules + Radix Themes を採用
  - tailwind 系は依存が増えてしまうので避けたい
    - tsx にクラス名が大量に入ると見通しが悪くなるのでこれも避けたい
    - 新し目の css の機能や animation を利用するときに、どうしても素の CSS を書く必要が出てきて、統一感がなくなるので避けたい
    - Chakra UI も同様に独自記法が気になる
  - emotion (CSS-in-JS) は、ランタイムのオーバーヘッドや、スタイルの分離が難しい点が気になるため避けたい
    - RSC では動かない
    - 独自記法になる
      - ゼロランタイム CSS-in-JS (Panda CSS) も同様
  - ピュアな CSS を使える UI ライブラリが良いと考える
  - typed-css-modules で型安全性も確保
  - 独自デザインシステムへ移行しやすいように component library は最低限の利用に抑える
  - Mantine も良さそうだが、今のところそこまで使いたいものが無いので、Radix Themes で最小限の UI コンポーネントを利用する形にする
    - 独自デザインシステムへ移行する際に Radix Primitives だけを残す形にしやすい
  - scss は不採用
    - 2026年2月現在、標準の css が変数、ネスト、計算などの機能をサポートしたため、 scss のメリットが薄れた
    - モジュール分割も CSS Modules で解決できる
    - mixin は react component として表現すれば代替可能だと考えられる
- testing-library
  - ユーザーの操作に近い形でテストが書ける
  - ブラックボックステストに近い形で、実装の詳細に依存しないテストが書ける
- happy-dom
  - jsdom より高速、軽量
